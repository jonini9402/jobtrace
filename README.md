# JobTrace (잡트)

> 취준생의 지원 현황을 한 곳에서 관리하는 서비스

🔗 배포 주소: [http://jobtrace.site](http://jobtrace.site)

---

## 프로젝트 소개

JobTrace는 취업 준비 과정에서 발생하는 지원 현황을 한눈에 관리하고, 데이터를 기반으로 병목 구간을 파악할 수 있도록 돕는 서비스입니다.

2년간 전업 취준생으로 지내면서, 기존 일정 관리 앱들이 취준생의 맥락(지원 상태 흐름, 마감일 관리, 합격률 추적 등)을 충분히 반영하지 못한다는 것을 직접 느꼈습니다. 단순히 공고를 정리하는 것에서 그치지 않고, **"나는 지금 어디에서 막히고 있는가"**를 데이터로 확인하고 행동으로 이어질 수 있도록 퍼널 구조로 설계했습니다.

---

## 주요 기능

### 1. 공고 관리
- 회사명, 직무, 공고 URL, 마감일, 플랫폼(사람인 / 원티드 / 점핏 등) 태그와 함께 공고 등록
- 지원 상태 관리: `관심` → `서류지원` → `코테` → `면접` → `최종합격 / 탈락`
- 상태 변경 시 날짜 자동 기록

### 2. 타임라인 & 캘린더 뷰
- 마감일 기준으로 정렬된 타임라인 뷰 제공
- 캘린더 뷰로 일정 한눈에 확인 가능
- D-day 자동 계산 (DB 저장 없이 프론트에서 계산)
  - D-3 이하 → 🔴 긴급
  - D-7 이하 → 🟠 주의
  - 나머지 → ⚪ 일반
- 상태 필터(전체 / 진행중 / 합격 / 탈락)로 원하는 공고만 조회 가능

### 3. 공고 상세 조회 및 메모
- 공고 클릭 시 상세 페이지에서 공고 정보, 지원 상태, 마감일, 공고 URL, 메모 확인
- 면접 후기, 준비 사항 등을 자유롭게 기록할 수 있는 메모 기능

### 4. 통계 / 인사이트
- 총 지원 수, 서류 합격률, 면접 전환율 등 핵심 지표 자동 계산
- 숫자 카드 + 차트(월별 추이, 상태별 분포)로 시각화
- 지원 수가 적을 경우 "지원율이 낮아요. 더 지원해보세요"와 같은 안내 멘트 제공

## 기술적 의사결정 & 고민의 흔적

### 칸반보드 → 타임라인 + 캘린더 뷰로 전환
처음에는 지원 상태를 칸반보드(관심/서류지원/코테/면접/최종합격 등 컬럼)로 보여주려 했습니다. 그런데 상태값이 10개에 가까워지면서 컬럼이 가로로 끝없이 늘어나, 한눈에 보기 위해 만든 보드가 오히려 한눈에 안 들어오는 문제가 생겼습니다.

다시 "취준생이 매일 들어와서 가장 먼저 확인하고 싶은 정보가 뭘까"를 고민했고, 결론은 **"이번 주 마감 임박 공고"**와 **"다음 면접 일정"**이었습니다. 그래서 상태를 컬럼으로 나누는 대신, 마감일 기준 타임라인 + 상태 필터 조합으로 구조를 바꿨습니다. 정보의 양이 아니라 "지금 당장 봐야 할 것"을 우선 노출하는 방향입니다.

### D-day를 DB에 저장하지 않은 이유
마감 임박 표시(🔴 D-3, 🟠 D-7)를 위해 D-day 값을 DB 컬럼으로 둘지, 매번 계산할지 고민했습니다. DB에 저장하면 매일 배치로 갱신해야 하는 부담이 생기고, 자정이 지나는 시점에 값이 어긋날 위험도 있었습니다. 마감일(고정값)만 저장하고, 화면에 보여줄 때 프론트에서 현재 날짜와의 차이를 계산하는 방식으로 결정했습니다. 데이터 정합성을 서버가 신경 쓰지 않아도 되는 구조를 택한 셈입니다.

### 통계 "해석" 기능을 단순 조건문으로 구현한 이유
기획 단계에서는 "서류 합격률 5% → 이력서 문제 가능성 높음"처럼 다양한 케이스별 해석 멘트를 고려했습니다. 하지만 범위를 넓히면 끝이 없고, 신입 개발자로서 핵심은 "숫자만 던지지 않고 행동을 유도하는 한 줄을 보여주는 경험" 자체였습니다. 그래서 우선 "지원 수가 5개 이하면 안내 멘트 표시"처럼 가장 명확한 조건 하나만 프론트에서 처리하는 방식으로 범위를 줄여 구현했고, 이후 케이스를 늘려갈 수 있는 구조로 남겨뒀습니다.

### StatusType을 enum이 아닌 static 상수 클래스로 설계
지원 상태값(`관심`, `서류지원`, `코테`, `면접`, `최종합격` 등)을 어떻게 관리할지 고민했습니다. enum은 타입 안정성이 좋지만, 상태가 늘어나거나 순서/그룹(예: "진행중" vs "종료" 묶음)을 유연하게 다루기엔 제약이 있다고 판단했습니다. 그래서 상태값과 관련 메타데이터(표시 순서, 그룹 등)를 함께 다루기 쉬운 static 상수 클래스 형태로 설계했습니다.

### 도메인 기반 패키지 구조
기능이 늘어날수록 계층형 패키지 구조(controller/service/repository를 최상위로 두는 방식)는 한 기능을 수정할 때 여러 패키지를 오가야 하는 불편함이 있을 거라 예상했습니다. 그래서 `공고`, `통계`, `유저` 등 도메인 단위로 패키지를 나누고, 그 안에 controller/service/repository를 두는 구조를 선택했습니다. 기능 단위로 코드를 찾고 수정하기 쉽게 만드는 것이 목표였습니다.

### Service 인터페이스 + Impl 분리
처음에는 Service 클래스를 바로 구현해도 충분하지 않을까 고민했지만, 인터페이스로 한 번 감싸두면 이후 테스트 코드 작성 시 Mock 객체로 대체하기 쉽고, 구현이 바뀌어도 호출하는 쪽 코드는 영향을 받지 않는다는 점에서 인터페이스 + Impl 패턴을 적용했습니다.

### recharts 대신 CSS 기반 차트
통계 페이지의 차트를 recharts로 구현하려 했으나, React 19와의 호환성 문제로 라이브러리 의존성에 문제가 생겼습니다. 외부 라이브러리에 발목 잡히기보다, 막대그래프 같은 단순한 형태는 CSS(width/height 비율 조정)로 직접 구현하는 방향으로 전환했습니다. 라이브러리 의존도를 낮추고 작은 단위부터 직접 만들어보는 경험을 택한 결정입니다.

## AI 활용 과정

이 프로젝트는 비전공자 출신 개발자로서 현업 기준을 이해하고 그에 맞춰 개발하기 위해 AI를 적극적으로 활용했습니다. 단순히 모르는 코드를 복사하는 방식이 아니라, **"현업에서는 왜 이렇게 하는가"를 먼저 이해하고 적용하는 과정**에 초점을 뒀습니다.

### 현업 기준 파악을 위한 질문
현장 경험이 없는 준비생으로서, 코드를 짜기 전에 "이 구조가 실제 현업에서도 쓰이는 방식인지"를 먼저 확인하는 습관을 들였습니다.

- 도메인 기반 패키지 구조가 현업에서 실제로 선호되는 이유
- Service 인터페이스 + Impl 분리가 왜 관습처럼 쓰이는지
- GlobalExceptionHandler와 ErrorCode enum을 활용한 에러 처리 방식이 실무에서 어떻게 쓰이는지
- JWT 필터 체인 구성 방식, SecurityConfig 설계 흐름

단순히 "이렇게 해라"는 코드를 받는 게 아니라, 먼저 왜 그런 구조를 쓰는지 물어보고 이해한 뒤 직접 구현하는 방식으로 진행했습니다.

### 에러 디버깅
배포 과정에서 발생한 에러들(EC2 디스크 부족, 메모리 부족으로 인한 Docker 빌드 실패, CORS 설정 누락, SQLIntegrityConstraintViolationException 등)을 AI와 함께 원인을 추적하고 해결했습니다. 단순히 해결책만 받는 것이 아니라, 에러가 왜 발생했는지 HTTP 상태 코드 의미부터 Spring 예외 처리 흐름까지 이해하면서 넘어가는 방식으로 학습했습니다.

## 로컬 실행 방법

### 1. 환경변수 설정
`src/main/resources/application-local.yml`의 환경변수를 본인 환경에 맞게 설정해주세요.

```yaml
spring:
  datasource:
    username: ${DB_USERNAME}   # 본인 MySQL 계정
    password: ${DB_PASSWORD}   # 본인 MySQL 비밀번호
jwt:
  secret: ${JWT_SECRET}        # 임의의 256비트 이상 문자열
```

### 2. 실행

```bash
# 빌드
./gradlew build -x test

# 실행
java -jar build/libs/jobtrace-0.0.1-SNAPSHOT.jar
```

> MySQL이 로컬에 설치되어 있어야 하며, `jobtrace` 데이터베이스가 생성되어 있어야 합니다.
